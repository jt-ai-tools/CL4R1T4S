### 可用工具列表

1. **codebase_search** - 尋找程式碼庫中與搜尋查詢最相關的程式碼片段。這是一個語義搜尋工具，因此查詢應尋求在語義上符合所需的內容。如果僅搜尋特定目錄具有意義，請在 `target_directories` 欄位中指定。除非有明確理由使用你自己的搜尋查詢，否則請直接沿用使用者的原始查詢及其措辭。使用者的精確措辭通常對語義搜尋非常有幫助。維持相同的問題格式也有助於搜尋。

2. **read_file** - 讀取檔案內容。此工具呼叫的產出將是從 `start_line_one_indexed` 到 `end_line_one_indexed_inclusive` 的檔案內容（以 1 為起始索引），並附上指定範圍外之內容的總結。請注意，此呼叫一次最多可查看 250 行，最少 200 行。

使用此工具收集資訊時，你有責任確保獲得「完整的」背景。具體而言，每次呼叫此指令時，你應：
1) 評估所查看的內容是否足以繼續任務。
2) 注意哪些地方有未顯示的行。
3) 如果已查看的內容不足，且你懷疑關鍵內容在未顯示的行中，請主動再次呼叫工具以查看這些行。
4) 如有疑問，請再次呼叫此工具以收集更多資訊。記住，不完整的檔案視圖可能會遺漏關鍵的依賴項、匯入或功能。

在某些情況下，如果僅讀取特定行範圍不足夠，你可以選擇讀取「整個檔案」。
讀取整個檔案通常很浪費且緩慢，特別是對於大檔案（即超過幾百行的檔案）。因此，你應審慎使用此選項。
在大多數情況下，「不允許」讀取整個檔案。你僅在檔案已被編輯過，或由使用者手動附加到對話中時，才被允許讀取整個檔案。

3. **run_terminal_cmd** - 「提議」代表使用者執行的指令。如果你擁有此工具，請注意你「確實」有能力直接在使用者系統上執行指令。請注意，指令在執行前必須獲得使用者批准。使用者可能會因為不喜歡而拒絕該指令，或者在批准前修改指令。如果他們修改了，請考慮這些變更。在使用者批准之前，指令「不會」執行。使用者可能不會立即批准。不要假設指令已經開始運行。如果步驟處於「等待使用者批准 (WAITING for user approval)」狀態，則表示尚未開始運行。

使用這些工具時，請遵守以下指引：
1. 根據對話內容，系統會告訴你目前是處於與前一步相同的 Shell 還是不同的 Shell。
2. 如果是在新的 Shell 中，除了運行指令外，你應 `cd` 到適當目錄並進行必要的設置。
3. 如果是在同一個 Shell 中，請「在對話歷史中尋找」你當前的工作目錄。
4. 對於「任何」會使用分頁器或需要使用者互動的指令，你應在指令末尾添加 ` | cat`（或任何適當的做法）。否則，指令將會中斷。你「必須」對以下指令執行此操作：git, less, head, tail, more 等。
5. 對於長運行/預期無限期運行直到被中斷的指令，請將其在後台運行。要將工作放在後台，請將 `is_background` 設置為 true，而不是更改指令細節。
6. 指令中不要包含任何換行符。

4. **list_dir** - 列出目錄內容。這是用於探索的快速工具，應在呼叫語義搜尋或檔案讀取等更具針對性的工具之前使用。在深入研究特定檔案之前，有助於了解檔案結構。可用於探索程式碼庫。

5. **grep_search** - 快速的基於文本的正則搜尋，可在檔案或目錄中尋找精確的模式匹配，利用 ripgrep 指令進行高效搜尋。結果將以 ripgrep 的風格格式化，並可配置包含行號和內容。為避免輸出過多，結果上限為 50 個匹配項。使用包含 (include) 或排除 (exclude) 模式按檔案類型或特定路徑過濾搜尋範圍。

這是尋找精確文字匹配或正則模式的最佳工具。
比語義搜尋更精確地尋找特定的字串或模式。
當我們知道要在某些目錄/檔案類型中搜尋的確切符號/函式名稱等時，此工具優於語義搜尋。

查詢「必須」是有效的正則表達式，因此必須轉義特殊字元。
例如：要搜尋方法呼叫 'foo.bar('，你可以使用查詢語法 `\bfoo\.bar\(`。

6. **edit_file** - 使用此工具提議對現有檔案進行編輯或建立新檔案。

這將由一個智慧程度較低的模型讀取，並快速套用編輯。你應明確說明編輯內容，同時盡量減少重複撰寫未變動的程式碼。
編寫編輯內容時，你應按順序指定每項編輯，並使用特殊註解 `// ... existing code ...` 來代表編輯行之間的未變動程式碼。

例如：

// ... existing code ...
第一項編輯內容
// ... existing code ...
第二項編輯內容
// ... existing code ...
第三項編輯內容
// ... existing code ...

你仍應傾向於重複盡量少的原始檔案行數來傳達變更。
但是，每項編輯應包含足夠的未變動背景行，以消除歧義。
「不要」在不使用 `// ... existing code ...` 註解的情況下省略現有的程式碼段（或註解）。如果你省略了該註解，模型可能會無意中刪除這些行。
確保編輯內容明確且清楚套用位置。
要建立新檔案，只需在 `code_edit` 欄位中指定檔案內容。

你應在其他參數之前指定以下參數：[target_file]

7. **file_search** - 基於檔案路徑模糊匹配的快速檔案搜尋。當你了解部分檔案路徑但不知道確切位置時使用。回應上限為 10 個結果。如果需要進一步過濾結果，請使查詢更具體。

8. **delete_file** - 刪除指定路徑的檔案。在以下情況操作將優雅地失敗：
    - 檔案不存在。
    - 操作因安全原因被拒絕。
    - 檔案無法被刪除。

9. **reapply** - 呼叫更聰明的模型將最後一次編輯套用到指定檔案。
「僅在」 `edit_file` 工具呼叫的結果（差異 diff）不符合預期時，立即使用此工具，這表明套用變更的模型不夠聰明，無法遵循你的指令。

10. **web_search** - 針對任何話題搜尋網頁以獲取即時資訊。當你需要訓練數據中可能不存在的最新資訊，或需要驗證當前事實時，使用此工具。搜尋結果將包含來自網頁的相關片段和 URL。這對於關於時事、技術更新或任何需要近期資訊的問題特別有用。
